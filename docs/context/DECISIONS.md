- D-20260501-001 | 競合価格の部屋タイプ別データは、指定なし response の後処理だけでは不足する。Chrome CDP 調査で、`/api/v5/competitor_prices` に `jalan_room_types[]=TWIN` を単独指定すると、指定なし response には含まれなかった TWIN plan が返ることを確認した。部屋タイプ条件として効く query は `jalan_room_types[]` であり、`jalan_facility_room_types[]`、`jalan_facility_room_type`、`room_types[]` は効かなかった。複数部屋タイプを同時指定しても各部屋タイプを網羅せず、指定集合内の最安値寄りに絞られるため、部屋タイプ別 snapshot は `SINGLE`、`DOUBLE`、`TWIN`、`TRIPLE`、`FOUR_BEDS` を単独 request として保存する | status: active | spec_link: docs/spec_001_analyze_expansion.md
- D-20260430-003 | `/api/v4/booking_curve` response 全体を localStorage に永続保存する旧 persistent cache は廃止する。Chrome CDP 確認では、localStorage 全体約 5.18 MB のうち `revenue-assistant:group-room-count:v4:<facility>:booking-curve:` 配下 36 件が約 5.16 MB を占め、`QuotaExceededError` の主因になっていた。同じ raw source は IndexedDB に保存済みであり、localStorage 側は memory cache と役割が重複するため、新規読み書きを止める。既存 key は facility prefix に限定して削除し、group-room result cache、booking curve raw source IndexedDB、derived reference curve IndexedDB、競合価格 snapshot IndexedDB は削除対象にしない | status: active | spec_link: docs/spec_001_analyze_expansion.md
- D-20260430-002 | 競合価格 snapshot は、Analyze 日付ページを開いた時点で保存候補にするだけでなく、利用者が Analyze 画面内で競合価格タブを開いた場合は、その stay_date の競合価格確認意図がより明確になったものとして取得優先度を上げる。ただし、booking_curve warm cache の queue に全日付分の競合価格取得を混ぜない。競合価格 snapshot は、現在開いている stay_date と現在の検索条件 signature に限定して、in-flight dedupe と連続取得抑制を維持した専用の保存状態として扱う。Analyze 画面の indicator には、booking_curve warm cache とは別に、競合価格 snapshot の保存状態、最終保存時刻、前回 snapshot 有無、保存失敗または競合施設未設定による skip を表示する | status: active | spec_link: docs/spec_001_analyze_expansion.md
- D-20260430-001 | 競合価格 endpoint は Analyze 日付ページで `GET /api/v5/competitor_prices` として呼ばれる。request には `x-requested-with: XMLHttpRequest` が必要で、query には少なくとも `date`、`min_num_guests`、`max_num_guests`、`yad_nos[]` が必要である。`meal_types[]` は省略可能で、省略すると素泊まり、朝食、夕食、朝夕食を含む広めの response を取得できる。`min_num_guests=1`、`max_num_guests=6`、`meal_types[]` 省略の request は `200` で取得できたが、response に実際に入る人数は該当プランが存在する人数だけである。`yad_nos[]` なしの全競合取得、宿泊人数範囲なしの取得、`max_num_guests=10` の広い取得は `400 BAD_REQUEST` になった。response は `own` と `competitors` を持ち、各 plan は人数、食事条件、プラン名、じゃらん部屋タイプ、URL、価格、自社価格との差分を持つが、在庫状態、販売停止、満室、ページング情報は持たない。初期 snapshot は、競合施設一覧と宿泊人数範囲を含む検索条件 signature ごとに保存する。競合施設は最大 5 件で後から入れ替え可能なため、保存時点の `yad_nos[]` と競合施設名を snapshot に保存し、施設単位の価格推移は `yad_no` ごとに追跡する | status: active | spec_link: docs/spec_001_analyze_expansion.md
